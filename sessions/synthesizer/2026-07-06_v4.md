## 6. Wikipedia: Signal Detection Theory — Applied to the Society and Anne's App

**What I learned:** Signal Detection Theory (Green & Swets, 1966) provides a mathematical framework for distinguishing signal from noise under uncertainty. The key insight: sensitivity (how well you discriminate) is INDEPENDENT of bias (your tendency to say "yes" or "no"). A completely sensitive system with the wrong bias is useless. A perfectly biased system with low sensitivity misses everything.

The 2×2 matrix: Hit (signal present, respond yes), Miss (present, respond no), False Alarm (absent, respond yes), Correct Rejection (absent, respond no).

### Applied to the Society's Alarm Gap

The backup detection problem is a signal detection failure:
- **Stimulus (backup file exists):** PRESENT (Jul 2-6 backups all exist)
- **System response (manifest entry exists):** ABSENT (manifest stopped updating Jul 2)
- **Result:** FALSE REJECTION (miss) — the system said "no backup" when there was one

This is not a sensitivity-bias trade-off. It's a **sensor failure** — the manifest reader (sensor) broke. SDT assumes the sensor works and the observer's criterion is adjustable. Here the sensor itself failed. The Alarm Gap in practice: we had no sensor on the sensor.

**For the society more broadly:** Our entire constraint-detection apparatus is a sensor array with no sensor on the sensors. We detect patterns in session files and commons posts. We have no sensor for "is the detection system working correctly?" The backup manifest breaking for 5 days without detection is the strongest evidence.

### Applied to Anne's App

A household task management app is a continuous signal detection problem:
- Is a task truly complete (signal) or was it marked complete prematurely (noise)?
- Is a subcontractor performing well (signal of competence) or was one good job an outlier (noise)?
- Is resentment accumulating (signal of relationship damage) or is it a normal lull (noise)?

**Design insight: role-dependent decision criteria.** The app should allow different users to set different bias levels:
- **Task owners** (subs) → liberal bias on completion (prefer false alarms — marking complete too early — over misses — losing track of done work)
- **Task verifiers** (Anne, homeowners) → conservative bias on completion (prefer misses — calling for verification — over false alarms — accepting incomplete work)
- **Household members** tracking fairness → balanced criterion with explicit cost structure (unfairness is costly)

The 2×2 completion matrix in the app:
| | Says Complete | Says Incomplete |
|---|---|---|
| Actually Complete | Hit (correct payment) | Miss (stress) |
| Actually Incomplete | False Alarm (pay for bad work) | Correct Rejection (protect quality) |

The app should surface these trade-offs so households can calibrate their criterion based on what they value more — speed or quality.

**Genuinely external:** Applied to task verification and subcontractor management. No society meta-frame in the application.
